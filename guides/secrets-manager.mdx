---
title: "Mount AWS Secrets Manager secrets in Kubernetes pods"
description: "Fetch secrets from AWS Secrets Manager and mount them as files in Kubernetes pods using the AWS Secrets Store CSI Driver Provider."
---

The AWS Secrets Store CSI Driver Provider (ASCP) can mount secrets stored in AWS Secrets Manager directly into your Kubernetes pods as files. When a pod starts, the ASCP fetches the specified secrets from Secrets Manager and presents them at the configured mount path inside the container.

## Prerequisites

- ASCP installed on your EKS cluster. See the [installation guide](/installation).
- An IAM role associated with your pod's service account via [IRSA](/auth/irsa) or [EKS Pod Identity](/auth/pod-identity).

## Required IAM permissions

Your pod's IAM role must allow the following actions on each secret you want to mount:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "secretsmanager:GetSecretValue",
        "secretsmanager:DescribeSecret"
      ],
      "Resource": [
        "arn:aws:secretsmanager:us-east-1:123456789012:secret:MySecret-??????"
      ]
    }
  ]
}
```

`DescribeSecret` is required in addition to `GetSecretValue` because the ASCP calls it during rotation reconciler checks to determine whether the mounted version is still current.

## Steps

<Steps>
  <Step title="Create a secret in Secrets Manager">
    Create the secret you want to mount. This example stores a JSON object with a username and password:

    ```bash
    aws secretsmanager create-secret \
      --region us-east-1 \
      --name MySecret \
      --secret-string '{"username":"memeuser","password":"hunter2"}'
    ```

    Note the secret ARN from the output — you can use either the friendly name or the full ARN in your `SecretProviderClass`.
  </Step>

  <Step title="Create an IAM policy and associate it with your service account">
    Create an IAM policy scoped to the secret, then attach it to the service account your pod will use.

    ```bash
    POLICY_ARN=$(aws --region us-east-1 \
      --query Policy.Arn --output text \
      iam create-policy \
      --policy-name my-app-secrets-policy \
      --policy-document '{
        "Version": "2012-10-17",
        "Statement": [{
          "Effect": "Allow",
          "Action": [
            "secretsmanager:GetSecretValue",
            "secretsmanager:DescribeSecret"
          ],
          "Resource": ["arn:*:secretsmanager:*:*:secret:MySecret-??????"]
        }]
      }')
    ```

    Then associate the policy with your service account using your chosen authentication method.
  </Step>

  <Step title="Create a SecretProviderClass">
    A `SecretProviderClass` tells the ASCP which secrets to fetch and how to present them. Set `objectType` to `"secretsmanager"` and `objectName` to the secret's friendly name or full ARN.

    <Tabs>
      <Tab title="IRSA">
        ```yaml
        apiVersion: secrets-store.csi.x-k8s.io/v1
        kind: SecretProviderClass
        metadata:
          name: my-app-aws-secrets
        spec:
          provider: aws
          parameters:
            objects: |
              - objectName: "MySecret"
                objectType: "secretsmanager"
        ```
      </Tab>
      <Tab title="EKS Pod Identity">
        ```yaml
        apiVersion: secrets-store.csi.x-k8s.io/v1
        kind: SecretProviderClass
        metadata:
          name: my-app-aws-secrets
        spec:
          provider: aws
          parameters:
            usePodIdentity: "true"
            objects: |
              - objectName: "MySecret"
                objectType: "secretsmanager"
        ```
      </Tab>
    </Tabs>

    Apply the manifest:

    ```bash
    kubectl apply -f secret-provider-class.yaml
    ```
  </Step>

  <Step title="Deploy a pod that mounts the secret">
    Reference the `SecretProviderClass` in your pod or deployment manifest by adding a CSI volume and a corresponding `volumeMount`:

    ```yaml
    apiVersion: apps/v1
    kind: Deployment
    metadata:
      name: my-app
    spec:
      replicas: 1
      selector:
        matchLabels:
          app: my-app
      template:
        metadata:
          labels:
            app: my-app
        spec:
          serviceAccountName: my-app-service-account
          volumes:
            - name: secrets-store-inline
              csi:
                driver: secrets-store.csi.k8s.io
                readOnly: true
                volumeAttributes:
                  secretProviderClass: "my-app-aws-secrets"
          containers:
            - name: my-app
              image: nginx
              volumeMounts:
                - name: secrets-store-inline
                  mountPath: "/mnt/secrets-store"
                  readOnly: true
    ```

    Apply the manifest:

    ```bash
    kubectl apply -f deployment.yaml
    ```
  </Step>

  <Step title="Verify the secret is mounted">
    Once the pod is running, confirm the secret file is present:

    ```bash
    kubectl exec -it \
      $(kubectl get pods -l app=my-app -o jsonpath='{.items[0].metadata.name}') \
      -- cat /mnt/secrets-store/MySecret
    ```

    You should see the raw secret value (the JSON string in this example).
  </Step>
</Steps>

## Object descriptor options

You can customize how each secret is fetched and named using the following fields in the `objects` list.

### Using a full ARN as objectName

When `objectType` is omitted, `objectName` must be a full Secrets Manager ARN. The ASCP infers the type from the ARN:

```yaml
objects: |
  - objectName: "arn:aws:secretsmanager:us-east-1:123456789012:secret:MySecret-AbCdEf"
```

### Controlling the mounted file name with objectAlias

By default, the secret is mounted as a file named after `objectName`. Use `objectAlias` to specify a different file name:

```yaml
objects: |
  - objectName: "MySecret"
    objectType: "secretsmanager"
    objectAlias: "database-credentials"
```

This mounts the secret at `/mnt/secrets-store/database-credentials` instead of `/mnt/secrets-store/MySecret`.

### Pinning to a specific version

Use `objectVersion` to pin to a specific version ID, or `objectVersionLabel` to pin to a version stage label. The most recent version is fetched by default.

```yaml
objects: |
  - objectName: "MySecret"
    objectType: "secretsmanager"
    objectVersionLabel: "AWSCURRENT"
```

Common version stage labels:

| Label | Description |
|---|---|
| `AWSCURRENT` | The current active version (default) |
| `AWSPREVIOUS` | The previous version, retained after rotation |

<Warning>
  Pinning `objectVersion` to a specific version ID requires updating the `SecretProviderClass` every time the secret rotates. Prefer `objectVersionLabel` or rely on the default behavior to always get the current version.
</Warning>

### Setting file permissions

Use `filePermission` to control the Unix file permissions of the mounted secret file. The default is `0644`. The value must be a four-digit octal string:

```yaml
objects: |
  - objectName: "MySecret"
    objectType: "secretsmanager"
    filePermission: "0600"
```
