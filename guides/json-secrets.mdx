---
title: "Mount individual keys from JSON secrets using JMESPath"
description: "Use JMESPath expressions to extract and mount individual key-value pairs from JSON-formatted AWS secrets as separate files in Kubernetes pods."
---

Many secrets stored in AWS Secrets Manager or SSM Parameter Store are JSON objects that contain multiple values — for example, a database credential secret might hold both a username and a password in a single secret. By default, the ASCP mounts the entire raw secret value as a single file. The `jmesPath` field lets you extract individual keys from the JSON and mount each one as its own file.

## How it works

When you add a `jmesPath` block to an object descriptor, the ASCP:

1. Fetches the full secret value from AWS.
2. Parses the value as a JSON document.
3. Evaluates each [JMESPath](https://jmespath.org/specification.html) expression against the parsed JSON.
4. Writes each result to its own file using the name specified in `objectAlias`.

Both the original secret file (containing the full JSON) and the individual key files are created in the mount directory.

## Example

Consider the following secret stored in Secrets Manager:

```json
{
  "username": "testuser",
  "password": "testpassword"
}
```

Without `jmesPath`, mounting `MySecret` creates a single file with the raw JSON string as its content. With `jmesPath`, you can mount `username` and `password` as separate files.

### SecretProviderClass with jmesPath

```yaml
apiVersion: secrets-store.csi.x-k8s.io/v1
kind: SecretProviderClass
metadata:
  name: my-app-json-secret
spec:
  provider: aws
  parameters:
    objects: |
      - objectName: "MySecret"
        objectType: "secretsmanager"
        jmesPath:
          - path: "username"
            objectAlias: "MySecretUsername"
          - path: "password"
            objectAlias: "MySecretPassword"
```

After mounting, the pod will have three files:

| File | Content |
|---|---|
| `/mnt/secrets-store/MySecret` | `{"username":"testuser","password":"testpassword"}` |
| `/mnt/secrets-store/MySecretUsername` | `testuser` |
| `/mnt/secrets-store/MySecretPassword` | `testpassword` |

## jmesPath entry fields

Each entry in the `jmesPath` list accepts the following fields:

| Field | Required | Description |
|---|---|---|
| `path` | Yes | The JMESPath expression to evaluate against the parsed JSON secret. |
| `objectAlias` | Yes | The file name under which the extracted value is mounted. |
| `filePermission` | No | A four-digit octal string (e.g., `"0600"`) for the mounted file. Defaults to the parent object's `filePermission`, which in turn defaults to `0644`. |

<Warning>
  Both `path` and `objectAlias` are required for every `jmesPath` entry. The ASCP will reject the `SecretProviderClass` at mount time if either field is missing.
</Warning>

## Handling hyphenated keys

JMESPath treats hyphens as subtraction operators. To reference a key that contains a hyphen, wrap the key name in double quotes, then wrap the entire value in single quotes in YAML:

```yaml
jmesPath:
  - path: '"hyphenated-key"'
    objectAlias: '"hyphenated-alias"'
```

For example, a secret with the shape `{"db-host":"db.example.com"}` requires:

```yaml
objects: |
  - objectName: "MySecret"
    objectType: "secretsmanager"
    jmesPath:
      - path: '"db-host"'
        objectAlias: "db-host"
```

## Setting per-file permissions

Use `filePermission` on individual `jmesPath` entries to restrict access to sensitive values:

```yaml
objects: |
  - objectName: "MySecret"
    objectType: "secretsmanager"
    jmesPath:
      - path: "username"
        objectAlias: "db-username"
        filePermission: "0644"
      - path: "password"
        objectAlias: "db-password"
        filePermission: "0600"
```

## Nested keys

JMESPath supports dot notation for nested objects. Given the following secret:

```json
{
  "database": {
    "host": "db.example.com",
    "port": "5432"
  }
}
```

Extract the nested values like this:

```yaml
jmesPath:
  - path: "database.host"
    objectAlias: "db-host"
  - path: "database.port"
    objectAlias: "db-port"
```

## Works with both Secrets Manager and Parameter Store

The `jmesPath` feature works with any `objectType`. For example, to extract keys from a JSON-formatted SSM parameter:

```yaml
objects: |
  - objectName: "/my/app/credentials"
    objectType: "ssmparameter"
    jmesPath:
      - path: "username"
        objectAlias: "app-username"
      - path: "password"
        objectAlias: "app-password"
```
